## 一、产品概述

### 1.1 产品背景
在教室、办公室、网吧、家庭聚会等场景中，用户希望在不接入互联网的情况下进行多人打字竞技。传统打字游戏需要互联网和中心服务器，限制了使用场景。本产品基于局域网自发现与直连通信技术，让任何支持网络的设备都能快速组队对战。

### 1.2 产品定位
一款纯局域网环境的多人实时打字竞技游戏，无需互联网、无需注册登录，设备自动发现并加入游戏，即开即玩。

### 1.3 核心价值
- **零网络依赖**：无互联网也可畅玩，适合内网环境
- **即开即用**：无需账号、无需配置，打开即创房/搜房
- **自动发现**：设备自动扫描局域网内房间，一键加入
- **低延迟**：局域网直连，延迟极低（<10ms）
- **跨平台**：电脑、手机、平板均可互联

### 1.4 目标用户
| 用户画像 | 场景描述 |
|----------|----------|
| 学校/培训机构 | 机房内组织打字比赛，无需外网 |
| 企业/办公室 | 午休团建、打字技能内训 |
| 家庭/聚会 | 多设备多人娱乐竞技 |
| 网吧/电竞馆 | 局域网对战活动 |

## 二、产品目标

### 2.1 业务目标（面向本地部署）
- 单房间支持最多8人同时游戏
- 设备从启动到进入房间 ≤ 10秒
- 游戏内操作延迟 ≤ 30ms

### 2.2 体验目标
- 无需任何网络配置，连上同一Wi-Fi/交换机即可互发现
- 界面清晰，无复杂登录流程
- 游戏过程流畅，无卡顿

### 3.1 功能架构图

```

│   └── 局域网扫描
├── 房间系统（无中心服务器）
│   ├── 创建房间（房主设备作为主机）
│   ├── 自动发现房间列表
│   ├── 直接IP加入（备选）
│   └── 房间内聊天（文本）
├── 游戏核心
│   ├── 难度选择
│   ├── 实时对战（帧同步/状态同步）
│   ├── 打字输入

│   └── 实时排名
├── 词库系统（本地存储）
```

### 3.2 设备发现与连接机制

#### 3.2.1 发现协议
- **传输层**：UDP（端口 25565 可配置）
- **广播地址**：`255.255.255.255` 或子网广播地址
- **发送频率**：房主每2秒发送一次“房间宣告”广播包
- **超时判定**：连续5次未收到宣告包，判定房间消失

#### 3.2.2 房间宣告包内容（JSON格式）
```json
{
  "type": "room_announce",
  "room_id": "abc123",
  "room_name": "速度与激情",
  "player_count": 3,
  "max_players": 8,
  "difficulty": "中等",
  "game_mode": "经典模式",
  "status": "waiting",  // waiting / gaming
  "host_ip": "192.168.1.101",
  "host_port": 25566
}
```

#### 3.2.3 加入流程
1. 客户端进入大厅后，持续监听UDP广播（端口25565）
2. 解析广播包，在“房间列表”中展示发现的房间
3. 用户点击“加入”，客户端通过TCP连接到房主IP+端口（25566）
4. 房主验证连接并分配玩家ID
5. 加入成功后，停止广播该房间的`player_count`递增

#### 3.2.4 备选加入方式
- **手动输入IP**：支持直接输入房主设备IP地址加入
- **二维码扫码**：房主可生成二维码（含IP+端口），其他设备扫码加入

#### 3.2.5 网络容错
- 房主断线：房主设备异常退出时，自动迁移房主给剩余玩家中ID最小者，游戏可继续（需支持状态同步）
- 玩家断线：显示“玩家已离开”，游戏继续，该玩家得分冻结
- 断线重连：30秒内重新连接可恢复（需保存玩家状态）

### 3.3 用户系统（简化）

#### 3.3.1 本地玩家标识
- **无需注册登录**
- 首次启动时生成唯一设备ID（UUID）存储在本地
- 玩家可自定义昵称（本地存储，默认“玩家+随机数”）
- 支持头像选择（内置头像库）

#### 3.3.2 本地数据记录
- 记录该设备上的历史战绩（总场次、胜场、最快WPM、最高准确率）
- 数据仅保存在本设备，支持导出/清除
- 成就系统（本地解锁，如“连击100次”等）

### 3.4 房间系统（主机模式）

#### 3.4.1 创建房间（房主端）
- 设置房间名称（默认“房主的昵称的房间”）
- 设置最大人数（2/4/6/8人）
- 选择游戏难度（简单/中等/困难/大师）
- 选择游戏模式（经典模式/限时模式/生存模式）
- 设置是否允许中途加入（默认不允许）
- 点击“创建房间”后，本机成为主机，开始UDP广播

#### 3.4.2 房间列表（客户端）
- 展示自动发现的房间，按信号强度/玩家数排序
- 房间信息：名称、人数、难度、模式、是否游戏中
- “游戏中”的房间不可加入（除非房主开启观战模式，V1.0暂不支持）
- 支持刷新按钮（强制重新扫描）

#### 3.4.3 房间内状态
- 房主可踢出玩家
- 所有玩家可切换准备状态
- 房主在所有玩家准备后开始游戏
- 房间内支持文字聊天（局域网内发送）

### 3.5 游戏核心机制（与互联网版一致）

#### 3.5.1 难度配置（同原版）

| 难度 | 词汇来源 | 单词长度 | 出现速度(词/秒) | 词汇总量 | 倒计时(秒/词) |
|------|----------|----------|-----------------|----------|---------------|
| 简单 | 基础词汇 | 3-5字母/2-3汉字 | 0.5 | 50 | 10 |
| 中等 | 常用词汇 | 4-7字母/3-4汉字 | 0.8 | 100 | 7 |
| 困难 | 进阶词汇 | 6-10字母/4-6汉字 | 1.2 | 150 | 5 |
| 大师 | 专业词汇 | 8-15字母/5-8汉字 | 1.8 | 200 | 3 |

#### 3.5.2 游戏流程
1. 房主点击“开始游戏”后，同步倒计时3秒给所有玩家
2. 服务端（房主设备）统一生成随机词块，同步下发
3. 各客户端本地输入，将输入结果上报房主
4. 房主进行判定、计分，并将最新得分广播给所有玩家
5. 达到结束条件（目标分数/时间到/生命耗尽）时，房主广播游戏结束
6. 展示排名与详细数据

#### 3.5.3 得分计算公式（不变）
**单次得分 = 基础分 × 速度系数 × 准确率系数 × 连击系数**

| 参数 | 计算公式 | 说明 |
|------|----------|------|
| 基础分 | 10 × 难度系数 | 简单1.0/中等1.5/困难2.0/大师3.0 |
| 速度系数 | max(0.5, min(2.0, 标准时间/实际用时)) | 标准时间=单词长度×0.3秒 |
| 准确率系数 | 1 + (当前正确率-0.8)×0.5 | 正确率低于80%时系数0.8 |
| 连击系数 | 1 + min(0.5, 连击数/100) | 最高1.5 |

#### 3.5.4 惩罚机制
| 错误类型 | 惩罚 |
|----------|------|
| 输入错误 | -3分，重置连击数 |
| 超时未输入 | -5分，刷新词块 |
| 连续错误3次 | 额外-10分，词块闪烁提示 |

#### 3.5.5 胜负判定
- **经典模式**：先达到目标分数（100/200/300分）者胜
- **限时模式**：60/120/180秒内得分最高者胜
- **生存模式**：每错3次扣除1生命（初始3），最后存活者胜

### 3.6 局域网同步架构

#### 3.6.1 网络拓扑
- **星型拓扑**：房主设备作为逻辑服务器，所有客户端直连房主
- **通信协议**：
  - 控制指令：TCP长连接（端口25566）
  - 实时得分/排名：TCP或UDP（可靠有序用TCP，实时性要求高可混用UDP+序列号）
- **推荐方案**：全部使用TCP（简化开发，局域网内延迟可接受）

#### 3.6.2 数据同步策略
| 数据类型 | 同步方式 | 触发条件 |
|----------|----------|----------|
| 房间状态 | 广播 | 玩家加入/离开/准备 |
| 词块生成 | 房主下发 | 按难度定时/玩家消除后 |
| 输入判定 | 客户端上报→房主广播 | 每次输入 |
| 得分更新 | 房主广播 | 每次判定后 |
| 排名变化 | 房主广播 | 每1秒或得分变化时 |
| 游戏结束 | 房主广播 | 达到结束条件 |

#### 3.6.3 防作弊（局域网环境）
- 房主拥有最终判定权，客户端仅作为输入终端
- 客户端上报输入内容及时间戳，房主重新计算耗时（防止客户端篡改时间）
- 若检测到异常（同一单词输入时间<0.1秒/字符），判为无效并警告
- 局域网内信任度较高，可适当放宽限制

### 3.7 词库系统（本地化）

#### 3.7.1 词库存储
- 所有词库文件随应用打包，存储于本地（JSON/SQLite）
- 支持用户导入自定义词库（.txt/.csv格式，每行一个词）
- 词库大小：各难度初始词库总量 ≥ 200个，后续可通过下载更新包扩充

#### 3.7.2 词库更新机制
- 无需联网时，用户可手动导入词库文件
- 若设备有互联网（可选），支持从官网下载最新词库包（非核心功能）

### 3.8 界面与交互（局域网优化）

#### 3.8.1 大厅界面
- 上方显示本机昵称、设备标识
- 左侧：“创建房间”按钮
- 中间：“房间列表”区域，自动刷新发现的房间
- 右侧：“手动加入”输入框（IP:端口）
- 底部：本机网络信息（IP地址、网络名称）

#### 3.8.2 房间内界面（同互联网版）
- 实时排名面板
- 当前目标词块（大字体居中）
- 输入框（自动聚焦）
- 剩余时间/目标进度
- 个人统计数据（准确率、速度、连击）
- 聊天框（局域网消息）

#### 3.8.3 设置界面（局域网特有）
- 修改本机昵称
- 选择UDP广播端口（默认25565，如冲突可修改）
- 选择TCP端口（默认25566）
- 清除本地战绩数据

### 3.9 数据统计（本地存储）

#### 3.9.1 个人数据指标
| 指标 | 定义 |
|------|------|
| 总场次 | 参与游戏总次数 |
| 胜场数 | 获得第一名次数 |
| 最快WPM | 历史最高每分钟正确输入单词数 |
| 最高准确率 | 单局最高准确率 |
| 最长连击 | 历史最大连续正确输入数 |

#### 3.9.2 成就系统（本地解锁）
| 成就名称 | 解锁条件 |
|----------|----------|
| 初出茅庐 | 完成1场游戏 |
| 十连胜 | 连续获得10场第一名 |
| 光速手指 | WPM达到100 |
| 百发百中 | 单局准确率100%且输入≥20词 |
| 房间满员 | 参与一次8人满员对局 |

## 四、非功能需求

### 4.1 性能需求
| 指标 | 要求 |
|------|------|
| 应用启动时间 | ≤ 3秒 |
| 房间列表刷新延迟 | ≤ 1秒（发现新房间） |
| 游戏内操作到显示结果 | ≤ 30ms（局域网内） |
| 单房间最大并发 | 8人同时游戏 |
| 房主设备CPU占用 | ≤ 30%（8人时） |

### 4.2 兼容性需求
| 平台 | 最低版本 | 分辨率 |
|------|----------|--------|
| Windows | 10/11 | 1366×768及以上 |
| macOS | 11+ | 同左 |
| Linux | Ubuntu 20.04+ | 同左 |
| Android | 8.0+ | 自适应 |
| iOS/iPadOS | 13+ | 自适应 |

### 4.3 网络要求
- 所有设备需处于同一IPv4局域网（同一子网）
- 支持Wi-Fi、有线网络混合组网
- 无需路由器开启特殊功能（UDP广播默认支持）

### 4.4 安全需求（局域网）
- 输入内容本地校验，防止注入（XSS过滤）
- 通信内容可简单校验（防止伪造包，但局域网内一般不要求高强度加密）
- 可选：支持设置房间密码（简单4位数字）

## 五、交互流程（局域网版）

### 5.1 房主流程

```
启动应用
    ↓
设置昵称（首次）
    ↓
点击“创建房间”
    ↓
配置房间参数（名称/人数/难度/模式）
    ↓
进入房间等待界面（开始广播）
    ↓
其他玩家加入（人数更新）
    ↓
所有玩家准备 → 房主点击“开始游戏”
    ↓
游戏对局（房主作为裁判+玩家）
    ↓
结算并展示排名
    ↓
返回房间 / 解散房间 / 再来一局
```

### 5.2 加入者流程

```
启动应用
    ↓
设置昵称（首次）
    ↓
大厅自动刷新房间列表
    ↓
选择房间 → 点击“加入”
    ↓
进入房间，点击“准备”
    ↓
等待房主开始游戏
    ↓
游戏对局（客户端上报输入）
    ↓
结算并展示排名
    ↓
返回大厅 / 留在房间
```

### 5.3 异常处理

| 异常场景 | 处理方式 |
|----------|----------|
| 房主退出游戏 | 自动迁移房主给剩余玩家中设备ID最小者，游戏继续；若游戏中退出，判定剩余玩家按当前排名结束 |
| 玩家断线 | 显示断线，保留得分，30秒内可重连；超时则移除 |
| UDP广播冲突 | 支持手动修改广播端口，或使用IP直连方式 |
| 防火墙拦截 | 提示用户允许应用通过防火墙（Windows需弹窗提醒） |

## 八、附录

### 8.1 局域网通信协议简要定义

**广播包（UDP）**：
- 目标端口：25565
- 格式：`{"type":"announce","room_id":"...","room_name":"...","player_count":3,"max_players":8,"difficulty":"中等","status":"waiting","host_ip":"192.168.1.101","host_port":25566}`

**TCP指令集**：
- `JOIN`：客户端请求加入
- `PLAYER_JOINED`：房主广播新玩家
- `PLAYER_READY`：玩家准备状态变更
- `GAME_START`：房主广播游戏开始
- `WORD_GEN`：房主下发新词块
- `INPUT_REPORT`：客户端上报输入
- `SCORE_UPDATE`：房主广播得分更新
- `GAME_END`：房主广播游戏结束

### 8.2 常见问题（FAQ）

| 问题 | 解答 |
|------|------|
| 为什么看不到其他房间？ | 检查所有设备是否在同一子网，防火墙是否允许UDP 25565端口 |
| 可以跨网段吗？ | 不支持，必须同一广播域（通常同一路由器下） |
| 支持多少设备同时在线？ | 单房间最多8人，多个房间可并存，总设备数受网络性能影响 |
| 数据会泄露到外网吗？ | 不会，所有通信仅在局域网内，无任何互联网请求 |

### 8.3 风险与应对

| 风险 | 概率 | 应对措施 |
|------|------|----------|
| 部分学校/企业防火墙限制UDP广播 | 中 | 提供IP直连和二维码扫码作为备选 |
| 房主设备性能不足 | 低 | 优化房主端计算量，建议房主使用性能较好的设备 |
| 无线网络丢包导致同步异常 | 中 | 使用TCP保证可靠传输，增加重传机制 |
